# SDD — Software Design Document

## Призначення документа
Цей документ описує архітектуру онлайн-платформи для навчання, її компоненти, взаємодію між ними та API.

## Загальна архітектура системи

Для реалізації онлайн-платформи для навчання обрано **багаторівневу клієнт-серверну архітектуру** з логічним поділом на frontend, backend та data layer. З точки зору архітектурного стилю рішення може бути реалізоване як **модульний моноліт**, оскільки для лабораторної роботи такий підхід є достатньо зрозумілим, зручним для опису та добре підходить для системи середньої складності.

Обрання модульного моноліту пояснюється такими перевагами:
- простіше розгортання у порівнянні з мікросервісною архітектурою;
- менша складність для невеликої команди;
- централізована бізнес-логіка;
- зручність тестування та супроводу;
- можливість подальшого виділення окремих модулів у сервіси, якщо навантаження зросте.

Узагальнена логіка роботи системи така:
1. користувач взаємодіє з frontend через браузер;
2. frontend надсилає запити до backend через REST API;
3. backend виконує бізнес-логіку та звертається до бази даних;
4. результат обробки повертається frontend і відображається користувачу.

## Основні компоненти системи

### 1. Authentication Module
Модуль відповідає за реєстрацію, вхід, вихід, управління сесіями, зміну пароля, перевірку токенів доступу та визначення ролі користувача.

### 2. User Management Module
Модуль зберігає профілі користувачів, інформацію про ролі, статуси облікових записів та адміністративні операції над ними.

### 3. Course Management Module
Модуль відповідає за створення, редагування, публікацію та каталогізацію навчальних курсів. Він працює з описами курсів, категоріями та структурою модулів.

### 4. Enrollment Module
Модуль реалізує функціонал запису студентів на курси, перевіряє доступність курсу, обмеження щодо ролі та зберігає факт підключення користувача до навчальної програми.

### 5. Content Delivery Module
Модуль забезпечує доступ до навчальних матеріалів: текстів, презентацій, файлів, відео або іншого вмісту, прив’язаного до конкретного курсу.

### 6. Testing Module
Модуль відповідає за зберігання тестів, варіантів відповідей, приймання результатів проходження та автоматичне оцінювання.

### 7. Reporting Module
Модуль формує звіти про результати студентів, прогрес проходження курсу, успішність та статистику для викладача й адміністратора.

### 8. Database Access Layer
Цей рівень інкапсулює доступ до бази даних, виконує збереження, оновлення, вибірку та транзакційну обробку даних.

## Взаємодія між компонентами

Взаємодія між компонентами відбувається відповідно до логіки сценаріїв користувача.

### Сценарій 1. Реєстрація та вхід
Користувач заповнює форму реєстрації у frontend. Дані передаються до Authentication Module, де виконується перевірка коректності, хешування пароля та створення нового запису в User Management Module і базі даних. Після цього користувач може виконати вхід, отримати токен доступу та перейти до роботи із системою.

### Сценарій 2. Створення курсу
Викладач через frontend створює новий курс. Запит надходить до Course Management Module, який перевіряє роль користувача через Authentication Module, зберігає структуру курсу і передає метадані до бази даних.

### Сценарій 3. Запис на курс
Студент переглядає каталог курсів через Course Management Module, після чого надсилає запит на запис. Enrollment Module перевіряє, чи має студент право доступу, чи курс відкритий для запису, та створює зв’язок між користувачем і курсом.

### Сценарій 4. Проходження тесту
Після завершення вивчення матеріалу студент відкриває тест. Testing Module надає набір питань, приймає відповіді, перевіряє їх, обчислює результат і передає його до Reporting Module для відображення в особистому кабінеті.

### Сценарій 5. Перегляд успішності
Викладач звертається до Reporting Module, який формує статистику на основі даних із Testing Module та Enrollment Module і повертає інформацію про прогрес студентів.

## Опис API / інтерфейсів

Нижче наведено приклад базового REST API для системи.

### API автентифікації
- `POST /api/auth/register` — реєстрація нового користувача
- `POST /api/auth/login` — вхід до системи
- `POST /api/auth/logout` — вихід із системи
- `GET /api/auth/me` — отримання інформації про поточного користувача

### API курсів
- `GET /api/courses` — отримання списку курсів
- `GET /api/courses/{id}` — перегляд детальної інформації про курс
- `POST /api/courses` — створення курсу
- `PUT /api/courses/{id}` — редагування курсу
- `DELETE /api/courses/{id}` — видалення або архівація курсу

### API запису на курс
- `POST /api/courses/{id}/enroll` — запис студента на курс
- `GET /api/users/{id}/courses` — перелік курсів користувача

### API навчальних матеріалів
- `GET /api/courses/{id}/materials` — отримання матеріалів курсу
- `POST /api/courses/{id}/materials` — додавання матеріалу викладачем

### API тестування
- `GET /api/tests/{id}` — отримання тесту
- `POST /api/tests/{id}/submit` — надсилання відповідей
- `GET /api/tests/{id}/result` — отримання результату

### API звітності
- `GET /api/results/me` — перегляд власних результатів студентом
- `GET /api/courses/{id}/results` — перегляд результатів курсу викладачем

## Логічна схема архітектури

Архітектуру системи можна подати у спрощеному вигляді:

**Користувач → Web Frontend → REST API → Backend Modules → Database**

де backend modules включають:
- Authentication Module
- User Management Module
- Course Management Module
- Enrollment Module
- Content Delivery Module
- Testing Module
- Reporting Module
