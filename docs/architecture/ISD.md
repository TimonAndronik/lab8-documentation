# ISD — Infrastructure Specification Document

## Призначення документа
Цей документ описує середовище розгортання, інфраструктурні компоненти та принципи масштабування онлайн-платформи для навчання.

## Середовище розгортання

Система передбачається для розгортання у хмарному середовищі або на віртуальному сервері з Linux-системою. Для забезпечення передбачуваності та відтворюваності розгортання використовується контейнеризація на базі Docker. Існує три основні середовища:
- **Development** — локальна розробка;
- **Staging / Test** — перевірка нових змін перед випуском;
- **Production** — робоче середовище для кінцевих користувачів.

У development-середовищі розробники запускають frontend, backend і базу даних локально. У staging-середовищі система проходить інтеграційне та системне тестування. У production-середовищі система повинна бути захищеною, доступною та стійкою до типових відмов.

## Основні інфраструктурні компоненти

### 1. Reverse Proxy / Web Server
Для приймання зовнішніх HTTP/HTTPS-запитів використовується Nginx. Він виконує термінацію TLS, маршрутизацію запитів до frontend і backend, а також може кешувати статичний контент.

### 2. Frontend Application
Клієнтська частина розгортається як окремий web-додаток, що надає користувачам графічний інтерфейс для роботи із системою.

### 3. Backend Application
Серверна частина розгортається як окремий контейнер або набір контейнерів. Вона відповідає за бізнес-логіку, обробку API-запитів та доступ до даних.

### 4. Database Server
Основною СУБД є PostgreSQL. Вона використовується для зберігання даних про користувачів, курси, записи на курси, результати тестів і журнали важливих дій.

### 5. File Storage
Навчальні матеріали можуть зберігатися або у файловій системі сервера, або в окремому об’єктному сховищі. Це дозволяє не перевантажувати базу даних великими файлами.

### 6. Cache Layer
Для прискорення повторних запитів і, за потреби, зберігання сесій може використовуватися Redis.

### 7. Monitoring and Logging
Для відстеження стабільності системи використовується централізований збір логів та моніторинг основних метрик: використання CPU, RAM, кількість помилок API, середній час відповіді, доступність.

### 8. Backup and Recovery
База даних повинна періодично резервуватися. Система повинна підтримувати відновлення після критичних збоїв на основі резервних копій.

## Принципи масштабування

Для забезпечення працездатності системи в умовах зростання навантаження передбачаються такі механізми масштабування:

1. **Горизонтальне масштабування backend**  
   За збільшення кількості користувачів можуть бути запущені додаткові екземпляри серверної частини.

2. **Балансування навантаження**  
   Перед backend-екземплярами може використовуватися балансувальник навантаження, який рівномірно розподіляє вхідні запити.

3. **Винесення зберігання файлів у окреме сховище**  
   Це зменшує навантаження на основний сервер і дозволяє гнучко керувати великими файлами.

4. **Використання кешування**  
   Найчастіше запитувані дані, наприклад каталог курсів, можуть кешуватися для прискорення відповіді.

5. **Масштабування бази даних**  
   При зростанні навантаження можливе вертикальне масштабування БД або впровадження read replicas для операцій читання.

## Нефункціональні аспекти інфраструктури

Інфраструктура повинна забезпечувати:
- захищене з’єднання по HTTPS;
- обмеження доступу до адміністративних інтерфейсів;
- централізоване логування;
- резервне копіювання;
- моніторинг здоров’я сервісів;
- швидке оновлення версій без тривалого простою системи.
